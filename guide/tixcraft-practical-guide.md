# 拓元售票實務研究與排錯指南

> 本文件整理拓元售票（TixCraft）的倒數開賣、釋票型態、常見跳頁原因與 HunterX 使用注意事項。內容分為官方可驗證資訊、合理技術推論與社群傳聞；傳聞不可視為固定規則。

## 官方可驗證資訊

- 拓元購票流程說明：正式開賣前，重新點選「立即購票」只會更新倒數；「立即訂購」按鈕於正式開賣後才會出現。各區剩餘數量僅供當下參考，因其他購票者也在訂購中，數量會持續變動。  
  來源：https://help.tixcraft.com/hc/zh-tw/articles/4404363906961
- 拓元選位方式說明：電腦配位依購買張數配位，剩餘空位不足或不連號時可能顯示「已無足夠數量」或「已無足夠連續座位」。  
  來源：https://help.tixcraft.com/hc/zh-tw/articles/4404363930129
- 拓元在 2026-04-07 聲明禁止使用購票程式、自動化工具或類似方式增加購票速度，並會依會員服務條款鎖定或停權異常帳號。  
  來源：https://tixcraft.com/news/detail/1143
- 拓元會員服務條款禁止以電腦程式、外掛、自動化設備或科技手段進行購票、干擾或操縱購票流程。  
  來源：https://tixcraft.com/terms-of-use

## 倒數開賣與 11:59:59.4 傳聞

社群常見說法是「中午 12:00 開賣時，在 11:59:59.4 點立即購票，剛好進入立即訂購」。這不是穩定規律，而是把本機點擊、前端處理、網路延遲與伺服器收到請求的時間差合併成一個猜測值。

實際判斷重點是伺服器收到請求的時間：

```text
伺服器收到請求時間 = 使用者點擊時間 + 瀏覽器處理時間 + 實際單向網路延遲 + 伺服器排隊延遲
```

因此，11:59:59.4 只有在整體延遲約 0.6 秒時才可能剛好落在 12:00:00 之後。若延遲較低，請求仍可能太早；若延遲較高，實際又會比別人慢。不同網路、瀏覽器負載、活動熱度與伺服器狀態都會改變結果。

較合理的實務做法：

1. 開賣前先完成登入、手機驗證、付款方式與活動頁確認。
2. 使用穩定網路與桌面瀏覽器，避免背景下載或大量分頁。
3. 接近整點才重新點選「立即購票」，避免過早連點導致未開賣狀態反覆刷新。
4. 不把固定毫秒數當成保證技巧；每場活動都應視作不同條件。

### 延遲校準刷新的安全做法

HunterX 不應實作「自動點擊立即訂購」或「自動送出購票請求」功能。若要改善指定時間操作，較安全且可控的做法是只校準「刷新觸發時間」：設定頁填官方開賣時間，HunterX 用本機校時與延遲估值換算實際刷新時間。

```text
觸發刷新時間 = 官方開賣時間 + 校時偏差 - 前端處理延遲 - 估計單向網路延遲 + 安全保留
```

建議時間來源與欄位如下：

| 欄位 | 估算方式 | 注意事項 |
|------|----------|----------|
| 校時偏差 | 優先使用 Windows/NTP 同步後的系統時間；Time.is 只作為觀察偏移的參考 | Time.is 或懸浮時鐘量到的是到該站的路徑，不等於到拓元的延遲 |
| 前端處理時間 | 用設定頁的本機 benchmark 估算事件排程、畫面幀/動畫等待、開始 reload 的耗時 | 不同電腦負載、瀏覽器分頁數會改變結果；不等於售票頁所有動畫 |
| 估計單向網路延遲 | 貼售票網址後，用少量 HTTP RTT 粗估，近似取 `RTT / 2` | 不是精準上行延遲；不高頻打拓元主站，避免造成異常流量 |
| 安全保留 | 固定保守值，避免太早抵達未開賣狀態 | 數值越大，越接近開賣後才觸發；數值越小，越容易太早 |
| 凍結秒數 | 開賣前 5-10 秒固定已算出的觸發時間 | 用來避免最後一秒數值跳動，不是保證吸收突發壅塞 |
| 伺服器排隊延遲 | 無法由使用者精準控制，只能透過歷史紀錄觀察 | 不建議把它當作可精準抵銷的數字 |

為什麼不直接用網路懸浮時鐘自動換算？因為它只能反映本機到該時鐘服務的校時差與路徑延遲，不代表本機到售票系統的實際單向網路延遲。最佳實務是：

1. 先把 Windows 系統時間用 NTP 同步到穩定狀態。
2. 在 `refresh_datetime` 填官方開賣時間，例如 `2026/07/09 12:00:00.000`。
3. 開啟延遲校準刷新，貼上售票網址後按「一鍵估算全部」，讓 HunterX 帶入校時偏差、售票站估計單向網路延遲與本機前端/動畫延遲。
4. 視風險調整安全保留；想保守就加大，想更貼近開賣邊界就降低。
5. 開賣前 5-10 秒凍結推算結果，避免最後階段因手動修改或量測抖動讓觸發點跳動。
6. 若最後一秒網路突然變慢，軟體無法在事件發生前知道；這種情況可能仍會太晚或太早，太早時頁面可能還是倒數狀態，需要人工再刷新一次。

這種模式能降低手動按秒的誤差，但不把 HunterX 變成違反平台規範的自動搶票流程。

## 零星釋票與大量放票

零星釋票通常是少量且不固定的庫存回流，常見來源包括：

- 使用者選到票後未在保留時間內完成付款或訂單確認。
- 付款失敗、訂單取消或重複訂購被取消。
- 退票完成審核後重新上架。
- 主辦方保留席、工程席、視線調整席或未售通路庫存回補。

大量放票通常是集中型庫存回補，常見來源包括：

- 主辦方公告釋出保留席、加開區域、加場或清票。
- 售票平台批次取消異常訂單後重新上架。
- 退票截止後主辦或平台集中整理可售座位。

拓元活動頁常見退票說明會寫明：在有退票釋出座位的前提下，完成退票作業後可能陸續將座位釋出並重新上架販售，退票截止日後也可能釋出座位。是否釋出、何時釋出與釋出數量仍以該活動公告及實際庫存為準。

## 驗證碼後跳回詳情頁

輸入驗證碼後跳回詳情頁，通常代表送出時伺服器已無法接受當前購票狀態，不一定是驗證碼錯誤。常見原因如下：

| 類型 | 說明 | 排查方式 |
|------|------|----------|
| 庫存競爭 | 你送出驗證碼前，區域或票數已被其他人完成訂購 | 降低票數、改選其他區域、觀察是否只在熱門區發生 |
| 多視窗 / 多實例 | 同帳號同活動多開，可能讓 session 或購票狀態互相覆蓋 | 同帳號同活動只保留一個主要購票視窗 |
| Token 過期 | 頁面停留過久、返回上一頁或複製分頁，導致 CSRF/captcha 狀態失效 | 重新從活動頁進入流程，不沿用舊分頁 |
| 伺服器負載 | 高流量時請求逾時、重導或回應不完整 | 記錄時間、URL、錯誤文字與是否大量使用者同時回報 |
| 異常操作判定 | 過快刷新、非人類操作節奏或多帳號同源請求可能被限制 | 降低刷新頻率，避免自動化與重複觸發 |

建議排錯紀錄欄位：

- 活動 URL、場次、區域關鍵字與票數。
- 跳回詳情頁前的最後 URL。
- 畫面是否曾出現「已無足夠數量」、「驗證碼錯誤」、「暫時鎖定」等訊息。
- 是否同帳號多視窗、多實例或不同裝置同時操作。
- HunterX 的 `instances` 面板是否有同一 profile 的 `name-2/name-3` 重複實例。

## HunterX 使用注意事項

- 多開不是越多越好。同帳號同活動多開會增加 session 互踢、跳回詳情頁與狀態錯亂風險。
- 上方 profile 叉叉是刪除設定檔；刪除前應先停止相關執行實例。新版流程會先停止相關實例，停不掉就不刪設定。
- `instances/default-2`、`instances/kktix-2` 這類資料夾是重複啟動實例的暫存狀態。離線且沒有對應設定檔時可以清理；還有新鮮 heartbeat 時不可刪。
- 針對拓元，刷新間隔不要過低。過度刷新會增加暫時鎖定、IP 風險與帳號異常風險。
- 本專案不應新增驗證碼繞過、反偵測、代理池、帳號池、排隊操縱或自動付款流程。
